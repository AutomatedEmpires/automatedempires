# Governance Fix Sequence

**Prepared:** 2026-07-12
**Scope:** Safe implementation order for provider ownership, recovery, cost controls, and transfer boundaries
**Current state:** Documentation only. The actions below are recommendations and approval gates, not evidence that any provider change has occurred.

## Guardrails

- Do not invite or remove users, change RBAC, rotate or revoke credentials, change billing or plans, buy domains, change DNS, activate mail, move money, deploy, or delete resources without separate approval for the named action.
- Preserve private recovery details, credential values, payment data, customer identities, and provider object identifiers outside this registry.
- Use a create/install/verify/observe/retire sequence for replacement access. Never remove the existing path before the replacement and rollback are proven.
- Treat repositories, provider projects, applications, tokens, folders, and configs as operational separation only. **Project separation is not independent ownership.**
- No step below establishes rollout-readiness or transfer-readiness. A phase closes only when its non-secret evidence is recorded and reviewed.

## Rollout gates versus transfer/sale gates

| Gate classification | Fix before this gate | Sequence coverage |
|---|---|---|
| **R — before rollout governance** | For each provider used by the venture: primary and backup admin/recovery, verified MFA, least privilege, current billing/alerts/caps/renewals, approved repository visibility, domain/mail/runtime authority, and tested backup/incident paths | Phase 1 documentation and GitHub visibility review; the applicable Phase 2 backup-admin work; Phase 3 billing controls. These are governance prerequisites, not a rollout-readiness verdict |
| **T — before transfer/sale** | Every applicable **R** item, plus legal/account/billing/data/domain/payment ownership, recipient-accepted administration and recovery, complete dependencies/retention, a transfer rehearsal, rollback, and post-transfer access removal | Phases 4 and 5, after the applicable Phase 1–3 work. Project separation alone never satisfies **T** |

The current classification is **not proven for R** and **not passed for T**. Product-specific rollout and security gates remain outside this sequence and may be stricter.

**Phase 5 exit evidence:** provider-by-provider recipient acceptance; tested recovery and rollback; signed legal/billing/data authority; continuity verification; post-transfer access-removal plan. Until all evidence exists, the pilot and every other venture remain not transfer-ready. This sequence makes no rollout-readiness claim.
